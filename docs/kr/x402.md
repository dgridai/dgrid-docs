# x402 API

**DGrid x402 API**는 DGrid가 제공하는 요청 단위 LLM 추론 결제 방식입니다. 개발자나 에이전트는 별도의 계정을 만들거나 충전 잔액을 준비할 필요 없이, **x402 결제 프로토콜**을 통해 API 요청 흐름 안에서 바로 추론 비용을 결제할 수 있습니다.

이 때문에 x402는 특히 다음과 같은 상황에 적합합니다:

* **에이전트 및 자동화 워크플로**가 추론 호출과 결제를 즉시 함께 처리해야 하는 경우
* **서버 측 도구와 스크립트**가 계정 개설이나 선충전 과정을 피하고 싶은 경우
* **온디맨드 통합**에서 "추론 요청, 결제 승인, 결과 수신"의 가장 짧은 경로를 원하는 경우

## 왜 x402가 중요한가

전통적인 AI 과금 방식은 보통 다음을 요구합니다:

* 공급자 계정 생성
* 선불 잔액 충전 또는 결제 수단 연결
* API 키, 쿼터, 청구 대시보드 관리

반면 DGrid의 x402 방식에서는 흐름이 훨씬 직접적입니다:

> 추론 요청 → 결제 요구사항 수신 → 서명 승인 → 결제와 추론을 함께 다시 요청

에이전트 시스템 입장에서는 이것이 큰 단순화입니다. 모델을 사용할지 결정하는 동일한 런타임이, 별도의 계정 관리 플로우로 이동하지 않고도 같은 과정에서 결제 승인과 정산까지 처리할 수 있기 때문입니다.

## 주요 사용 사례: 에이전트가 직접 추론 비용 지불

x402는 특히 다음과 같은 흐름을 위해 설계되었습니다:

1. 에이전트가 모델을 선택
2. 에이전트가 추론 API 호출
3. 에이전트가 해당 추론 비용 결제
4. 에이전트가 결과를 받아 다음 작업 수행

이 방식은 다음에 특히 유용합니다:

* 백엔드 워크플로 안에서 동작하는 AI 에이전트
* 결제 인식형 추론이 필요한 Web3 에이전트 및 프로토콜
* 추론을 직접 결제 가능한 리소스로 취급하고 싶은 다단계 자동화

## 동작 방식

DGrid의 x402는 **2단계 요청 플로우**를 사용합니다:

1. **첫 번째 요청은 `x-payment` 없이 전송**
   서버가 `402 Payment Required` 와 서명 가능한 payment requirements 를 반환합니다.
2. **클라이언트가 서명 후 재요청**
   클라이언트가 `x-payment` 페이로드를 생성하고, 같은 추론 요청을 결제 헤더와 함께 다시 보냅니다.

이 패턴은 결제 승인을 명시적으로 유지하면서도 전체 추론 호출을 자동화 친화적으로 유지합니다.

## 지원 결제 자산

현재 x402 엔드포인트는 다음을 사용합니다:

* **프로토콜**: x402
* **네트워크**: `eip155:56` (BSC)
* **결제 토큰**: `USD1`
* **자산 컨트랙트**: `0x8d0D000Ee44948FC98c9B98A4FA4921476f08B0d`

## 운영 메모

정산이 성공하면 서비스는 백엔드의 `x402_payment_logs` 테이블에 결제 로그를 기록합니다.

## x402와 AI Gateway를 언제 사용할까

다음이 필요하면 **AI Gateway** 를 사용하세요:

* 전통적인 API 키 기반 접근
* 통합 계정 기반 모델 라우팅
* 표준 과금 방식의 장기 제품 통합

다음이 더 중요하면 **x402** 를 사용하세요:

* 요청 단위 직접 결제
* 선충전 잔액이 필요 없는 접근 방식
* 에이전트가 한 플로우 안에서 추론과 결제를 함께 처리하는 방식
