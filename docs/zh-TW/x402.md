# x402 API

**DGrid x402 API** 是 DGrid 提供的按次支付模型推理方式。開發者或智能代理不需要先註冊帳號、預先充值或管理獨立計費流程，而是可以直接在 API 請求流程中，透過 **x402 支付協議** 同步完成 LLM 推理費用支付。

這讓 x402 特別適合以下場景：

* **Agent 與自動化工作流**，需要一邊調用模型、一邊即時完成付款
* **伺服器端工具與腳本**，不希望再額外處理帳號開通與充值流程
* **按需整合**，希望以「發起推理、授權付款、取得結果」的最短路徑接入能力

## 為什麼 x402 很重要

傳統 AI 計費流程通常需要：

* 建立供應商帳號
* 預先充值餘額或綁定支付方式
* 管理 API 金鑰、配額與計費後台

而在 DGrid 的 x402 模式下，流程更直接：

> 發起推理請求 → 收到付款要求 → 簽名授權 → 帶付款資訊重新請求並取得結果

對 agent 系統來說，這大幅簡化了推理調用。做出模型調用決策的同一個 runtime，也能在同一流程中完成付款授權與結算，而不需要跳入額外的帳號管理流程。

## 主要應用場景：Agent 直接支付推理費用

x402 主要面向這類需求：

1. Agent 選擇模型
2. Agent 發起推理 API 請求
3. Agent 完成該次推理付款
4. Agent 使用回傳結果繼續執行任務

這對以下場景很有價值：

* 後端工作流中的 AI Agent
* 需要具備支付感知能力的 Web3 Agent 或鏈上協議
* 希望將推理視為可直接結算資源的多步驟自動化流程

## 流程如何運作

DGrid 的 x402 使用 **兩步式請求流程**：

1. **第一次請求不帶 `x-payment`**
   服務端返回 `402 Payment Required` 與可簽名的 payment requirements。
2. **客戶端簽名後重新請求**
   客戶端生成 `x-payment` 載荷，並攜帶該標頭再次發送相同推理請求。

這種模式使付款授權保持明確，同時仍然讓整個推理調用維持簡單且適合自動化。

## 支援的支付資產

目前 x402 端點使用：

* **協議**：x402
* **網路**：`eip155:56`（BSC）
* **支付代幣**：`USD1`
* **資產合約**：`0x8d0D000Ee44948FC98c9B98A4FA4921476f08B0d`

## 運維補充

結算成功後，服務端會將付款記錄寫入後端的 `x402_payment_logs` 表。

## 何時使用 x402，何時使用 AI Gateway

若您需要以下能力，請使用 **AI Gateway**：

* 傳統 API Key 存取方式
* 統一帳戶型模型路由
* 標準帳戶計費的長期產品整合

若您更需要以下能力，請使用 **x402**：

* 請求級別的直接付款
* 無需預充值帳戶餘額
* 讓 agent 在同一流程中完成推理與支付
